# Auto Update Release Flow

Fuzitter is configured to distribute the Windows installer through GitHub Releases and check for updates at startup from the same release feed.

## One-time setup

1. Confirm the GitHub repository is `Negi310/Fuzzy`.
2. Make sure the default branch contains `.github/workflows/release.yml`.
3. Distribute only the installer generated by electron-builder.

## Release steps

1. Update the version in `package.json`.
2. Commit the version bump.
3. Create and push a tag that matches the version.

Example for version `0.1.2`:

```powershell
git add package.json package-lock.json
git commit -m "Release v0.1.2"
git tag v0.1.2
git push origin main
git push origin v0.1.2
```

## What happens after the tag push

The GitHub Actions workflow will:

1. Install dependencies with `npm ci`.
2. Run `npm run release`.
3. Build these files:
   - `Fuzitter-Setup-x.y.z.exe`
   - `latest.yml`
   - `Fuzitter-Setup-x.y.z.exe.blockmap`
4. Publish them to the GitHub Release for that tag.

## What to distribute to users

Send users only:

- `Fuzitter-Setup-x.y.z.exe`

Do not send the unpacked `Fuzitter.exe` from `dist/win-unpacked` or old folder builds.

## How startup updates work

1. The user installs Fuzitter with `Fuzitter-Setup-x.y.z.exe`.
2. On startup, the installed app checks GitHub Releases for a newer version.
3. If a newer version exists, it downloads the update automatically.
4. The update is applied when the app restarts.

## Verification checklist

- The installed app shows the current version in Settings.
- A newer release tag exists on GitHub.
- The GitHub Release contains `latest.yml`, the installer, and the blockmap.
- The app downloads the update after launch and prompts for restart or applies it on quit.
